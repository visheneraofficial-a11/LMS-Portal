5-Layer System Architecture
Click each layer to expand. Each layer depends on the one above it — data flows top-down.
1
TENANT — Multi-Tenant Root
▼

The Tenant is the top-level container. Each coaching institute is a tenant. All data is isolated per tenant — ABC Institute cannot see XYZ Institute's students.

Key Fields: code, name, subdomain, custom_domain, plan_type (FREE/BASIC/PREMIUM/ENTERPRISE), youtube_api_key, status

Example: code: ABC → abc.enfclass.com → contains 156 students, 12 teachers, 5 admins

Admin Location: Sidebar → TENANTS → Tenants

Every other model in the system has a tenant FK that links back here. This is the isolation boundary.

2
RBAC — Roles, Permissions & Access Control
▼

Role-Based Access Control defines who can do what. Three components:

Permission — an atomic action: students.read.batch = "read students in your batch"

Role — a bundle of permissions: "Academic Coordinator" = 12 permissions

RolePermission — the junction: Role X has Permission Y

Permission Format: module.action.scope — e.g., attendance.update.tenant

Actions: CREATE, READ, UPDATE, DELETE, EXECUTE, APPROVE, EXPORT

Scopes: OWN → BATCH → BRANCH → TENANT → GLOBAL (escalating access)

Role Inheritance: A role can have a parent_role — it inherits all parent permissions plus its own

Admin Location: Sidebar → ACCOUNTS → Roles / Permissions / Role permissions

3
USERS — Admin, Teacher, Student, Parent
▼

Four user types, all extending a common BaseUser (UUID PK, email, phone, MFA, status):

Admin — 6 admin types (Super → Tenant → Branch → Academic/Finance/Support). Has FK to Role + permissions_override JSON.

Teacher — YouTube integration, batch assignments, inline permissions (can_override_attendance, etc.). Self-FK reporting_to for hierarchy.

Student — FK to Batch, class level, exam target, fee status, notifications. Core user — everything else references students.

Parent — FK to Student (primary_student), relationship type, toggle flags for what they can view.

Admin Location: Sidebar → ACCOUNTS → Students / Teachers / Admins / Parents

4
ACADEMIC STRUCTURE — Sessions, Subjects, Batches
▼

Defines what is taught, when, and how it's organized:

AcademicSession → "2025-2026" (time boundary)

Group → "Main Campus" (organizational division)

Category → "JEE Wing" (sub-division under Group)

Subject → "Physics" → Chapter → "Mechanics" → Topic → "Newton's Laws"

Batch → "NEET-2025-A" (links to Session + Group)

BatchStudent = Student enrolled in Batch

BatchTeacher = Teacher assigned to Batch + Subject

Plus reference tables: Language, State, City, Religion, School

Admin Location: Sidebar → ACADEMICS (15 items)

5
OPERATIONS — Classes, Tests, Attendance, Communication
▼

Daily activities that reference users and academic structure:

Classes: ScheduledClass (teacher + batch + chapter + YouTube), ClassAccessToken, ClassWatchTime

Assessments: Test → TestSection → Question → TestAttempt → TestAttemptAnswer, TestFeedback, OfflineTestMarks

Attendance: Mark present/absent/late per student per batch per day

Communication: SupportTickets, Announcements, DirectMessages, Notifications

Materials: StudyMaterial (PDF/video), PhotoGallery, Scholarship, TopperStudent

Monitoring: UserDevice, UserSession, LoginHistory, UserActivity, AuditLog

Config: SystemSettings, FeatureFlags, MFAPolicy, MaintenanceWindow, AIFeatureConfig

Roles & Permissions System
How RBAC controls access across the admin panel.

Admin Type Hierarchy

Click each level to see details. Higher = more authority.

SUPER_ADMIN — Full System Access
Access: Everything. All tenants, all settings, all data.
Example Person: Platform Owner / CTO
managed_branches: N/A (all)
Typical Permissions: *.*.global
Can create: Other admins, tenants, system settings
TENANT_ADMIN — Full Tenant Access
Access: Everything within their tenant. Cannot see other tenants.
Example Person: Institute Director
managed_branches: All within tenant
Typical Permissions: *.*.tenant
Can create: Branch admins, teachers, students, batches
BRANCH_ADMIN — Branch-Level Access
Access: Only data in their managed_branches[] array.
Example Person: Campus Head
managed_branches: ["Main Campus"]
Typical Permissions: *.*.branch
Can create: Teachers, students within their branch
ACADEMIC_ADMIN
Access: Academic modules only — Subjects, Batches, Tests, Chapters
Example Person: Academic Coordinator
Cannot access: Financial data, system settings, audit logs
FINANCE_ADMIN
Access: Financial modules — Fee status, scholarships
Example Person: Accounts Manager
Cannot access: Academic content, system settings
SUPPORT_ADMIN
Access: Support modules — Tickets, Communication
Example Person: Help Desk Lead
Cannot access: Academic or financial data

Permission Scope Escalation

Each scope includes everything below it. Higher scope = wider access.

OWN
Only records you created/own
→
BATCH
Records in your assigned batches
→
BRANCH
Records in your managed branches
→
TENANT
All records in your tenant
→
GLOBAL
All records across all tenants

Role Inheritance Example

Child roles automatically inherit all parent permissions.

📛 Teacher (Parent Role)
Level: 30 · Type: SYSTEM · Applies To: TEACHER
classes.read.own attendance.read.batch students.read.batch
↓ inherits ↓
📛 Head Teacher
Level: 80 · parent: Teacher
classes.read.own attendance.read.batch students.read.batch + attendance.update.batch + students.export.tenant + tests.approve.batch
Total: 3 inherited + 3 own = 6 permissions
📛 Junior Teacher
Level: 50 · parent: Teacher
classes.read.own attendance.read.batch students.read.batch + materials.create.own + tests.create.own
Total: 3 inherited + 2 own = 5 permissions

Permission Code Examples

📗 CREATE Permissions
students.create.branch tests.create.own batches.create.tenant materials.create.own

Create new records. Scope limits where the record can be assigned.

📘 READ Permissions
students.read.batch attendance.read.branch audit.read.global tests.read.tenant

View existing records. Most common permission type.

📕 DELETE Permissions
students.delete.own tests.delete.branch audit.delete.global

Destructive action — usually limited to higher scopes. Often requires APPROVE prerequisite.
